# Contributing / 参与贡献

欢迎通过 Issue 提交使用反馈、错误案例、兼容性信息、来源补充和 RFC 建议。为保护用户隐私和项目未来的双许可能力，请先遵守以下边界。

## 不要提交

- 真实 Personal OS、`POS.md`、个人目标、聊天记录、平台数据或其他私人资产；
- 凭证、Token、密钥、内部路径和未脱敏日志；
- 付费内容、第三方全文或无法确认再分发权利的材料；
- 直接从个人工作目录复制出的测试夹具；
- 让 Agent 获得更大默认权限、永久删除或绕过审批的变更。

测试与示例必须使用虚构数据和新建的临时目录。

## 提交流程

1. 先搜索已有 Issue 和 RFC；涉及核心语义、兼容性、安全或许可的改动，先提交 RFC/Issue。
2. 说明问题、用户场景、预期行为、替代方案和验收方式。
3. 保持 `SKILL.md` 精简，把细节放入一层 `references/`；不要让普通任务全量加载新文档。
4. 运行 `npm run validate`，并确认 `npm pack --dry-run` 不含私有材料。
5. 在 PR 中列出风险、测试证据、文档变化和是否影响已有 Personal OS 数据。

## 贡献许可与商业再许可

本项目需要保留为公开版本和商业版本提供不同许可的权利。Issue 中的想法和事实反馈不会自动转移版权；但代码、模板、Skill 指令、原创文档或图示等实质性 PR，在合并前需要贡献者完成项目维护者提供的单独 Contributor License Agreement（CLA），其中应包含在公开许可和商业许可下再许可该贡献所需的授权。

在 CLA 流程完成前，维护者不应合并实质性第三方贡献。仅使用 DCO 或“inbound = outbound”不足以保证项目可以向商业客户提供完整的商业例外。

贡献者同时保留其贡献的版权，并可继续按 CLA 允许的方式使用自己的贡献。CLA 不应改变用户个人内容的权利，也不能授权贡献者本来无权提交的第三方材料。

> CLA 的最终文本与签署流程应在接受第一项实质性外部贡献前经过适用司法辖区的专业法律审查。本文件不构成 CLA，也不构成法律意见。

签署记录、法律身份信息和商业合同应保存在仓库之外的受控系统中；`.gitignore` 会拒绝常见的本地 `cla-signatures/`、`commercial/` 和 `legal/private/` 工作目录。

## 安全问题

不要在公开 Issue 中披露可直接利用的安全细节、个人路径或数据样本。先创建不含敏感细节的安全联系请求，等待维护者建立私密沟通方式。
