# RFC 0001：Inbox 作为统一入口

- 状态：accepted
- 日期：2026-07-17
- 影响：路由、文件结构、用户体验

## 问题

传统文件系统要求用户在行动前先知道文件属于哪个目录。对于 Agent 新手，这增加了开始成本；对于已有大量 Agent 产物的用户，不同 Agent 又会按各自理解创建新目录。

## 决策

`00_Inbox` 同时接收未分类文件和自然语言任务，是 Personal OS 的统一入口。Router 先框定目标和交付物，再选择 intent、主要 Area/Project、Context 与执行 Agent。

Router 输出建议，不直接修改正式文件。缺失信息会改变主要归属、交付物、持久化、风险或外部动作时，最多先问一个最必要的问题；只影响可撤销草稿时，说明假设并进入隔离 Run。

## 为什么不让模型直接猜路径

同一主题可能用于学习、创作、产品、投资或临时问答。仅按关键词分类会得到多个同样合理但后果不同的路径，错误写入会把自动化成本转回用户维护成本。

## 替代方案

- **用户先选文件夹**：准确但认知成本高，不适合小白。
- **每个 Agent 固定一个 Inbox**：实现简单，但跨 Agent 形成新的孤岛。
- **每次全库扫描后自由判断**：上下文成本高，仍缺少稳定分类契约。

## 后果

- 用户可以从同一位置开始；
- Router 需要明确分类契约和置信/缺失信息输出；
- Inbox 必须定期处理，不能成为永久 Resources；
- 路由与正式提交必须分开，避免“判断即写入”。

## 验收

- 明确请求无需追问即可进入匹配 Project/Area；
- 同一句概念问题在不同用途下会得到不同归属；
- 低置信且影响正式归属时会追问；
- Router 不直接修改正式资产；
- 常规路由先查询索引，不递归加载全库。
