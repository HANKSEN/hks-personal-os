# RFC 0003：隔离 Run 与 Changeset 写入协议

- 状态：accepted
- 日期：2026-07-17
- 影响：安全、Agent 权限、CLI、恢复

## 问题

把 Personal OS 根目录的读写权限交给 Agent 可以显著降低维护成本，但模型和宿主都可能误判路径、覆盖正式资产、执行导入内容中的指令或在多步操作中只完成一半。

## 决策

语义工作与正式文件变更分为两个阶段：

1. Agent 在自身 `99_AI/hosts/<host-id>/runs/<task-id>` 内读取授权上下文并自由生成草稿；Host 代表实际执行产品，Role 只是本次任务能力配置；
2. Agent 用结构化 Changeset 声明 `create/update/move/archive/trash`；
3. CLI 校验显式根、路径、作用域、策略、哈希和受保护 Context；
4. 用户预览并明确批准；
5. CLI 建立快照，事务应用，失败时恢复，成功后记录审计并刷新索引；
6. Undo 在没有后续冲突时恢复指定事务。

系统不提供永久删除，Router 也不直接写正式空间。

## 替代方案

- **完全只读 Agent**：安全但维护成本仍由用户承担，无法实现授权协作。
- **Agent 直接编辑 + Git 回滚**：简单，但不保证目标目录是 Git 仓库，也无法阻止路径越界和部分事务。
- **每次人工复制粘贴**：控制明确，但破坏低摩擦目标，且难以审计批量关系变更。

## 后果

- 正式写入比普通文件编辑多一个提案步骤；
- CLI 必须保持确定性、离线和零模型依赖；
- Undo 不是备份，用户仍需独立备份和恢复验证；
- trusted 模式也不能突破显式根目录、导入内容边界和永久删除禁令。

## 验收

- 未加批准标志时 Apply 只预览、不产生正式变更；
- 越界、绝对路径、符号链接、过期哈希和未授权 Context 修改被拒绝；
- 任一操作失败后所有受影响路径恢复到事务前状态；
- 后续编辑与 Undo 冲突时默认拒绝覆盖；
- Apply、拒绝和 Undo 都留下结构化审计事件。
