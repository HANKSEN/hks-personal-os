# RFC 0005：批次级审批、撤销与大型文件原样复制

- 状态：accepted
- 日期：2026-07-20
- 影响：Changeset、审批、Undo、迁移、诊断

## 问题

真实归档任务可能超过单次可审阅的操作数量，或包含不适合进入文本差异和聊天传输层的大型文件。过去 `taskId` 同时承担任务身份和 Undo 身份，导致同一任务的第二批变更无法独立应用；更严重的是，碰撞后的清理路径可能误删第一批已经提交的历史。

## 决策

1. `taskId` 只表示用户任务与 Agent Run；可选 `changeId` 表示一批不可变、可独立 Undo 的变更。
2. 未提供 `changeId` 时仍使用 `taskId`，兼容既有 Changeset 与历史。
3. `.pos/history/` 与事务 seal 以 `changeId`（即返回的 `undoId`）为键；失败清理只能删除本次尝试实际创建的目录。
4. 保留每批最多 25 项的可审阅边界，超限时按逻辑拆批并分配不同 `changeId`，不通过无限提高上限规避人工审阅。
5. 大型暂存文件使用 `mode: "opaque-copy"` 原样创建。预览绑定文件大小和 SHA-256，但不读取完整正文、不生成完整 diff，也不改变压缩格式。
6. 交互面板超时保持提案为 `awaiting_approval`，允许重新打开，不把超时解释为拒绝或取消。
7. Doctor 交叉检查已应用审批与 Undo 历史，缺失时明确报错。

## 安全后果

- 每批审批与回滚边界更小，多个批次不会共享历史目录。
- 大型文件仍必须来自当前 Run 的 `proposed/`，仍受根目录、符号链接、写入范围、大小和哈希校验约束。
- `opaque-copy` 首版只支持 `create`，不允许借此覆盖现有文件。
- Undo 仍不是独立备份；缺失的历史不会被系统伪造修复。

## 验收

- 同一 Task 的两个不同 `changeId` 可以分别 Apply 和 Undo。
- 重复 `changeId` 被拒绝，既有 manifest 和 seal 字节不变且仍可 Undo。
- 超过 32 MiB 的合成文件可以原样复制，目标哈希与暂存源一致。
- 大型单行文本不会形成超大 MCP 响应。
- 面板超时后正式资产不变，原 proposal 可以重开。
