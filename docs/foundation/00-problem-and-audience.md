# 问题与受众

## 要做什么

Hks Personal OS 是一套本地优先、Agent 原生的个人信息操作协议。它把普通文件夹、Markdown 上下文、可移植 Skill 与 Skill 内置的确定性运行时组合起来，使用户能够从一句自然语言或一个文件开始，由 Agent 协助完成意图澄清、上下文路由、任务执行、正式写入和经验沉淀。全局 CLI 只是可选入口。

## 解决的问题

现有个人文件管理和 Agent 协作之间存在五个断点：

1. 用户需要先知道“去哪个文件夹”，才能开始工作；小白往往在第一步就卡住。
2. Agent 能生成大量产物，却缺乏统一归属、生命周期和正式版本规则。
3. 收藏、理解、经历、方法和交付物混在一起，检索结果无法表达资产意义。
4. 每次协作都重新解释背景，或为了避免遗漏而把整个仓库塞入上下文。
5. Agent 获得文件权限后可能误写、覆盖、越界或执行导入内容中的恶意指令。

Personal OS 因此把“自然语言入口”“文件组织”“上下文检索”“安全写入”和“经验复利”设计成同一条链路，而不是五套互不相连的工具。

## 面向谁

| 人群 | 起点 | 主要问题 | 首要价值 |
|---|---|---|---|
| Agent 新手 | 没有文件体系，也不清楚 Agent 应如何授权 | 不知道从哪里开始，担心配置和误操作 | 一个 Inbox 入口、15 分钟首次闭环、安全默认值 |
| 已有 Agent 经验但文件混乱 | 已产生很多代码、文章和分析文件 | Agent 自建目录、重复文件、正式稿难找 | PARA 归属、统一上下文、隔离 Run 和可审阅提交 |
| 长期创作者与研究者 | 有稳定输入与输出 | 数据、复盘、SOP 和作品没有形成闭环 | Artifact、Data、Experience、Principle 之间可追踪 |
| 个体开发者与知识工作者 | 同时经营多个领域与项目 | 每个工具都维护一份背景，上下文难复用 | 本地源文件、多 Agent 可移植 Skill、按需检索 |
| 高要求团队或商业使用者 | 希望嵌入内部工作流 | 需要审计、权限边界和可控集成 | 确定性 CLI、Changeset、商业许可与定制空间 |

## 核心使用场景

- 把长文、PDF 或链接放入 Inbox，明确是临时阅读、长期学习、创作素材还是项目研究。
- 从模糊问题开始，经过一次最小追问后进入正确 Area 或 Project。
- 完成文章、视频、代码、Skill 或方案，并把正式版本、数据和复盘分开保存。
- 把一次行动记录为 Experience，把经重复验证的规律升级为 Principle 或 SOP。
- 在投资、职业等高风险问题中分离来源、事实、AI 推断、个人判断和实际结果。
- 让不同 Agent 使用同一套目录与协议，而不是各自创建无法互认的工作区。

## 必要性

- Agent 的生成成本快速下降，整理、判断正式版本和保持长期上下文的成本反而上升。
- 只靠聊天历史难以形成可携带、可审计、可由不同 Agent 使用的个人资产。
- 只靠传统 PARA 能回答“当前放哪里”，但不能完整回答“这是外部资料、我的理解、一次经历、可复用规则还是已发布成果”。
- 给 Agent 根目录写权限而没有事务、审批和恢复机制，会把低认知成本转化为高数据风险。

## 可行性

- Markdown、JSON 和普通目录可被用户直接阅读，不依赖专有数据库。
- PARA 提供足够简单的物理骨架，五类资产提供稳定的语义边界。
- Agent Skill 负责模糊意图和语义判断，CLI 负责路径、事务与审计，避免要求模型同时承担确定性职责。
- 本地索引可以重建；常规任务只加载核心、领域、项目和少量相关资产，无需扫描全库。
- v1 不包含 GUI、云同步、向量数据库、多 Agent Mesh 和自动外部执行，因此可用零依赖 Node.js 本地运行时实现并验证。

## 成功定义

系统成功不是“自动把每个文件都分对”，而是满足以下结果：

- 用户无需先理解目录即可开始；仅在错误路由代价明显时回答一个最小问题。
- 正式资产有唯一主要归属，跨领域关系通过链接而不是复制表达。
- Agent 的默认草稿不会直接污染正式空间，正式修改可预览、可审计、可撤销。
- Knowledge、Experience、Principles、Artifacts 和 Data 可被人和 Agent 一致区分。
- 随使用次数增加，用户重复解释背景和重新发明工作流的次数下降。

## v1 边界

v1 不在既有非空知识库中原地改名、移动或删除；只支持独立备份提醒后的只读诊断，以及把已审项目复制到全新 Personal OS。它不替用户做外部发布、交易、支付或通信，不提供永久删除，也不把模型输出当成事实或用户价值观。系统辅助定义问题和提出变更；用户仍负责重要判断、授权、备份和最终确认。
