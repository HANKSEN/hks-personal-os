# 设计原则

这些原则用于解决功能冲突。若新增功能与其冲突，应先通过 RFC 修改原则，而不是悄悄增加例外。

## 1. 先降低开始成本

用户可以从根目录或 `00_Inbox` 表达意图，不必先选择 Agent、Area、Project 或文件类型。系统承担初步理解和推荐成本。

## 2. 先问目的，再判断位置

同一句“帮我了解 Agent”可能服务于学习、创作、产品或投资。路由先判断期望变化和交付物，再判断领域与持久化位置。

## 3. 只做最小必要澄清

缺失信息会改变主要归属、交付物、持久化、风险或外部动作时才追问。只影响可撤销草稿的歧义，应说明假设并在隔离 Run 内继续。

## 4. PARA 管物理归属，资产类型管语义

Projects、Areas、Resources、Archive 回答“内容现在与行动的关系”；Knowledge、Experience、Principles、Artifacts、Data 回答“长期保存的资产意味着什么”。两个维度不互相替代。

## 5. 一个正式副本，一个主要所有者

每项正式资产只有一个规范副本和一个主要 Area 或 Project。跨领域关系使用稳定 ID、元数据和链接表达，避免复制导致版本分叉。

## 6. 工作过程与正式资产分离

研究、草稿和中间输出留在 Project 或实际宿主的 `99_AI/hosts/<host-id>/runs`；只有通过审阅的结果才进入正式资产目录。Archive 表示生命周期结束，不表示垃圾。

## 7. 人决定意义，AI 降低维护成本

人负责价值、目标、重要判断和最终授权。AI 可以提出、起草、关联、整理甚至在已授权范围内写入，但不能把推断悄悄升级为用户事实、目标或原则。

## 8. 语义控制面与确定性数据面分离

Skill/Agent 处理模糊语言、任务框定和内容生成；CLI 处理路径约束、哈希、事务、审计和撤销。不能让“模型觉得安全”代替程序校验。

## 9. 渐进式加载上下文

常规任务按 Core → Area/Project → 相关资产加载。索引是可重建缓存，不是事实来源；找不到相关个人背景时明确报告缺口，不编造记忆。

## 10. 默认可逆，越高风险越显式

草稿默认隔离；正式写入先预览；核心 Context 需要额外批准；删除采用软删除；外部高风险动作不在 v1 自动执行。备份始终高于 Undo 和 Git。

## 11. 本地文件是可迁移的事实来源

系统不把用户锁定在特定笔记软件、模型或 Agent 宿主。即使 CLI 和索引不可用，用户仍能用普通文件工具阅读和迁移自己的资产。

## 12. 规范保持小，扩展通过 Context

根结构和核心术语保持稳定；每个 Area、Project 和 Agent 可以用一份 Context/Manifest 定义自己的状态、工作流和规则，避免把所有人的生活强制成同一棵目录树。
