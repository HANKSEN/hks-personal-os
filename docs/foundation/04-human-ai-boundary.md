# 人机协作边界

Personal OS 不把“AI 能写”与“AI 能决定”混为一谈。权限取决于空间、任务范围和风险，而不是一条绝对禁令。

## 职责分工

| 对象 | 主要职责 | 不应擅自做的事 |
|---|---|---|
| 用户 | 定义目标、价值和重要约束；确认关键事实；批准高风险或正式变更 | 把备份、安全和最终判断完全外包给 Agent |
| Router/Orchestrator | 框定任务、最小澄清、选择主要归属、装配上下文和 Agent | 直接写正式资产、伪造置信度或用户意图 |
| 专业 Agent | 在授权上下文内研究、创作、实现或复盘 | 扩张任务范围、执行未授权外部动作 |
| Reviewer | 检查证据、冲突、路径、完成条件和变更风险 | 用审阅意见代替用户的价值判断 |
| Skill 内置运行时 / 可选 `pos` CLI | 确定性校验、预览、事务写入、审计和 Undo | 调用模型、推断自然语言意图或发送个人数据 |

## 三类空间

### AI 自治空间

`99_AI/hosts/<host-id>/runs/<task-id>` 内，实际执行该任务的 Agent 可以在当前任务范围内创建、修改和清理中间文件。不同宿主的临时内容相互隔离；失败应留存为可检查状态，不能影响正式资产。

### 协作空间

Projects、Areas、Resources 和 Archive 允许 AI 在用户授权下协助写入。默认流程仍是提案 → Changeset → 预览 → 批准 → Apply。

### 受保护 Context

`POS.md` 和 Area/Project `CONTEXT.md` 可以由 AI 提议修改，也可以在用户认同后由 CLI 写入，但需要额外的 protected-content 批准。AI 不得把行为推断直接改写成用户身份、价值或目标。

## 权限模式

| 模式 | 正式空间 | 适合场景 |
|---|---|---|
| `safe` | 只读；只在 AI Run 写草稿 | 初次体验、未知 Agent、只做探索 |
| `collaborative` | Changeset + 明确批准后写入 | 默认日常使用 |
| `trusted` | 可按预先声明的任务范围提高自动化 | 熟悉协议、已有备份和审计习惯的用户 |

即使在 `trusted` 模式，路径也不能越过显式根目录，导入内容不能改变权限，v1 也不允许永久删除或自动执行高风险外部动作。

## 最小追问规则

如果“用途不同会导致正式归属不同”，Router 应先问一个面向结果的问题。例如：

> 你希望这篇长文主要用于临时读懂、沉淀长期知识，还是支撑当前文章/项目？

如果答案只影响草稿标题或临时目录，则说明假设并继续，不打断用户。

## Agent 独立工作区

Agent Manifest 应声明：

- purpose 与触发条件；
- 可读取的 Context 类型；
- 可写入的 Run 路径；
- 预期输出和完成条件；
- 需要的批准与禁止动作。

Agent 的长期可靠性来自共享协议和正式资产，不应依赖某个 Agent 私有、不可见且无法迁移的“记忆文件夹”。
