# 验收方法

Personal OS 的效果需要同时从可用性、路由质量、资产质量、上下文效率和文件安全五个维度验证。单纯“Agent 能生成文件”不构成通过。

## 验收矩阵

| 维度 | 核心问题 | v1 可验证指标 |
|---|---|---|
| 上手成本 | 小白能否不理解目录也开始？ | 按文档 15 分钟内完成 Init、Knowledge 提案、Apply、Undo |
| 路由质量 | 模糊请求是否进入正确主要归属？ | 黄金案例中的 intent/owner/output 与期望一致；必要歧义会追问 |
| 资产边界 | 五类资产是否可被一致区分？ | 创作、SOP、阅读、投资案例没有把 Resource 当 Knowledge 或把单次复盘当稳定 Principle |
| 上下文效率 | 是否避免全库扫描和上下文膨胀？ | 正常 Context 请求最多返回设定数量和字符预算；10,000 文件合成库仍受边界约束 |
| 写入安全 | Agent 是否只能在授权范围变更？ | 路径逃逸、符号链接、过期哈希、未批准 Context、硬删除全部拒绝 |
| 可恢复性 | 失败和误操作能否恢复？ | 每种 Changeset 操作均覆盖 Apply、故障回滚和冲突感知 Undo |
| 可移植性 | 是否绑定某个 Agent？ | Skill/CLI 可由通用目录、Codex、Claude 或显式自定义 Skill 目录安装 |
| 隐私边界 | 发布包是否混入私有材料？ | 包内容清单不含 `specs/`、个人数据、绝对用户路径、凭证和运行产物 |

## 测试层级

1. **结构校验**：检查必需文件、版本一致性、许可范围和发布白名单。
2. **单元测试**：路径、策略、Markdown、索引、Context 和 Changeset 原语。
3. **集成测试**：本地运行时进程、Setup、Init、Index、Run、Apply、Doctor、Undo、Audit 与复制迁移。
4. **安全测试**：遍历、符号链接、Prompt Injection、历史篡改和事务故障。
5. **工作流测试**：长文阅读、公开创作、SOP-to-Skill、投资研究与复盘。
6. **前向评估**：由未参与实现的 Agent 在全新合成根目录中按照 Skill 完成任务。

## 测试数据原则

- 所有自动化测试只使用操作系统临时目录和虚构内容；
- 测试根有唯一标记，缺少标记时拒绝读写和清理；
- 不用用户现有 Personal OS 做自动化复制、迁移或“试跑”；所有审计与迁移测试只使用临时目录中的合成夹具；
- 测试前后校验根外 canary，证明没有成功越界写入；
- 失败夹具在必要时保留以便检查，但不得进入发布包。

## 发布门槛

一个版本只有在以下条件全部满足时才可以宣称通过：

- `npm run validate` 全部成功；
- 版本号、插件 Manifest、README 和 Changelog 一致；
- `npm pack --dry-run` 不包含私有 Spec、个人资产、凭证或运行数据；
- 文档中的安装、安全和许可说明与实际行为一致；
- 已知限制明确记录，没有把兼容目标表述为已验证事实；
- 若许可发生变化，旧版本不可撤销的授权边界已明确说明。

## 用户效果验证

技术测试之外，建议用 7 天真实但低风险的试运行观察：

- 从输入到开始工作的平均步骤是否减少；
- 需要手动纠正路由或移动文件的比例；
- 找到正式版本所需时间；
- 每次任务加载的上下文文件数和字符数；
- 产生的 Experience 中有多少形成了可验证 Principle 候选；
- 用户是否理解并实际使用备份、预览和 Undo。

这些指标用于发现系统是否真的降低认知成本，而不是鼓励 Agent 创建更多文件。
