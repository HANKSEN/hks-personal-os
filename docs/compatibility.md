# Compatibility

The file protocol uses Markdown, JSON, and JSONL and is independent of a specific note application.

The embedded deterministic runtime targets Node.js 20 or later and does not require a database or external npm dependency. The v1 validation evidence in this repository was produced on macOS; Linux and Windows should be treated as portability targets until their CI matrices are added.

The canonical Skill follows the Agent Skills format. Verified native filesystem targets include Codex, Claude Code, OpenClaw, and Hermes Agent. WorkBuddy and CodeBuddy use packaged plugin manifests plus a shared Agent Skills fallback. TRAE, TRAE SOLO, and other hosts without a sufficiently stable public private-directory contract use the shared `~/.agents/skills/personal-os` target or an explicit `--skill-dir`.

Every install result distinguishes `verified-native`, `verified-plugin-compatible`, `verified-standard`, and `standard-fallback`. The shared location is an interoperability option, not a guarantee that every host discovers it. Undocumented product paths are never guessed. See [Agent compatibility](agent-compatibility.md).

The host Agent must support reading the Skill and invoking local Node.js files. Natural-language clarification and routing occur in that host model. The embedded runtime is model-independent and does not make network or LLM calls. If a host cannot discover Skills but can explicitly read instructions and execute the runtime, label it compatibility mode.

Interactive approval is an optional capability layer. Codex desktop native MCP form elicitation is not a supported decision surface for Personal OS: current host rendering may flatten or normalize the message body and make exact file plans hard to review. In Codex, the Skill must use the conversation-native approval visual generated by `approval-visual`. The card controls its own table and section layout while its buttons only send a proposal-bound follow-up decision. The Agent must then re-read the persisted approval and verify its full digest before calling `decide`.

The installer detects locally configured Codex and Claude Code CLIs and can register the bundled stdio MCP server by default. In Codex this MCP server provides preview, proposal state, integrity validation, execution, and a fail-closed inline-visual handoff; it deliberately does not open the native approval form. The CodeBuddy/WorkBuddy plugin bundle declares the same local MCP runtime through its documented plugin-root variable. Another host that supports MCP form elicitation and preserves a reviewable layout may render a native confirmation panel. Clients without either interface receive an exact proposal-ID text fallback.

At runtime, the invoking product should pass a stable Host ID so temporary work is isolated under `99_AI/hosts/<host-id>/runs/`. Codex, Claude Code, and other products are Hosts; `research`, `creator`, `builder`, and `reviewer` are separate task Roles. Unknown adapters use `generic` rather than guessing identity from user content.

Git is optional. Personal OS also keeps task-scoped rollback snapshots under `.pos/history/` so a formal change remains reversible without Git.

New-root initialization deliberately refuses an existing non-empty target on every supported platform. Existing directories use a separate read-only audit and reviewed copy-to-new-root migration path; in-place takeover remains unsupported.
