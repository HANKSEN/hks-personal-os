# 使用前的备份、安全提示与免责声明

[English](safety.en.md) | 简体中文

Personal OS 通过明确根目录、有限上下文、隔离 Run、Changeset 预览、审批、审计和 Undo 降低 AI Agent 操作文件的风险，但不能消除所有风险。

## 使用前先备份

在首次授权 Codex、Claude Code、WorkBuddy、QCode、Kimi 或其他 Agent 读取、整理、移动、修改任何有价值的目录前，建议完成一次独立备份：

1. 备份完整目录，而不只是 Markdown 文件；图片、附件、代码、数据库、配置和隐藏文件都可能重要。
2. 将备份保存在 Personal OS 根目录之外，例如外置磁盘、另一台设备、可信云盘快照或独立备份服务。
3. 不要只确认“备份任务成功”，应随机恢复几个文件，确认备份确实可读取。
4. 重要工作建议保留至少两个独立副本，并根据变化频率建立定期备份。
5. 在扩大 Agent 权限、批量整理、升级工具或修改策略前，再创建一次新快照。

Personal OS 的 `.pos/history/`、Undo、Git 提交、云同步和服务商版本历史都能帮助恢复，但它们可能与原目录一起损坏、同步错误或被错误操作，因此不应作为唯一备份。

## 最小授权原则

- 只授权明确的 Personal OS 根目录，不要默认授权整个主目录、磁盘或文件系统根目录。
- 初次使用保持 `collaborative` 模式，让正式写入先预览、再批准。
- 检查 Changeset 的操作类型、来源、目标、范围和 diff；看不懂时不要批准。
- 修改 `POS.md`、`CONTEXT.md`、目标、价值观和长期规则时，确认这是自己的真实判断。
- 不要把密码、Token、私钥、助记词或敏感身份资料放进 Agent 可读取目录。
- 外部文档、网页和 Inbox 内容可能包含 Prompt Injection，应始终作为不可信数据处理。
- 发布、发消息、付费、安装第三方软件或交易等外部动作需要单独确认，不属于普通文件审批。

## 不同 Agent 的风险不同

不同宿主的沙箱、终端权限、插件、MCP、自动批准规则、记忆机制和网络权限并不相同。Personal OS 无法控制或保证：

- 宿主 Agent 或模型始终正确理解用户意图；
- 第三方插件、Skill、MCP、Shell 命令或集成没有缺陷；
- 操作系统、同步软件、云服务或存储设备不会失败；
- 用户批准的范围和命令一定符合真实目标；
- Undo 能恢复独立于 Personal OS 之外的外部动作。

使用新的 Agent 或提高自动化权限前，应重新检查它实际获得的读写、网络和执行权限。

## 项目边界

- 默认安装器只安装 Skill 及其内置运行时；全局 CLI 是可选项。软件安装授权不会自动授权初始化、读取或迁移个人数据目录。
- 新建旅程只初始化全新或空目录；初始化先在同级临时目录中完成健康检查，非空目录会转入已有目录旅程。
- 已有目录旅程只在独立备份提醒和精确读授权后进行只读审计；默认迁移把已审项目复制到独立的新 Personal OS，不在源目录原地改名、移动或删除。
- 审计和迁移发现源目录变化、路径重叠、符号链接、凭证候选、哈希失配或目标冲突时会停止，而不是猜测或覆盖。
- Personal OS 不提供永久删除动作，但宿主 Agent 仍可能拥有系统级文件工具，因此仍应限制授权范围。
- 本项目不自动发布内容、发送消息、安装生成的 Skill、执行支付或进行投资交易。
